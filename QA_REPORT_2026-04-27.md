## 1 ‚Äî Onboarding / Login

Files: [login.tsx](app/(auth)/login.tsx), [forgot.tsx](app/(auth)/forgot.tsx), [reset.tsx](app/(auth)/reset.tsx), [verify-email.tsx](app/(auth)/verify-email.tsx), [_layout.tsx](app/(auth)/_layout.tsx), [app/index.tsx](app/index.tsx), [lib/session.tsx](lib/session.tsx)

### P3 ‚Äî Deep-link reset code dedup is in-memory only ‚Äî `app/_layout.tsx:26`
**Repro:** Tap reset link in email ‚Üí app cold-starts and exchanges code ‚Üí user backgrounds and the OS kills the app ‚Üí tap link again from email.
**Cause:** The `exchangedCodes` Set guarding against double-exchange lives in JS module memory and is reset on cold launch. A second cold launch with the same `?code=` re-attempts exchange.
**Fix:** Persist consumed codes (e.g. an `AsyncStorage` Set with a 24h TTL). Currently low risk because the second exchange fails gracefully and routes to forgot screen with `err=expired` ‚Äî annoying UX but no security or data issue.

**‚úì Auth ‚Äî 1 P3.**

## 5 ‚Äî Add/Remove ·Éõ·Éù·Éú·Éê·É¨·Éò·Éö·Éî·Éî·Éë·Éò (Bottom Sheet)

Files: [components/CrewSection.tsx](components/CrewSection.tsx), [components/BottomSheet.tsx](components/BottomSheet.tsx)

### P2 ‚Äî Bottom sheet dismisses before parent's async `onChange` resolves ‚Äî `components/CrewSection.tsx:58-62`
**Repro:** Tap "·É®·Éî·Éú·Éê·ÉÆ·Éï·Éê" on the crew member form on a slow connection.
**Cause:** Sheet calls `dismiss()` immediately and the parent runs `onChange(...)` in the background. If `onChange` rejects (server error, optimistic rollback), the sheet is already gone and the user sees no error context ‚Äî only the row reappearing/disappearing in the list.
**Fix:** Add a local `submitting` flag; keep the sheet open with a spinner while `onChange` resolves; only `dismiss()` on success. Toast on error stays usable.

### P3 ‚Äî Crew member ID generation can collide on `Math.random()` fallback ‚Äî `components/CrewSection.tsx:210-213`
**Repro:** On a device where `crypto.randomUUID` is unavailable, add two crew members in the same millisecond.
**Cause:** Fallback combines `Date.now()` + `Math.random()`, which is unique enough in practice but theoretically collidable. Real risk is very low ‚Äî `expo-crypto` polyfills randomUUID, and humans don't add two members in the same ms.
**Fix:** Either drop the fallback (rely on `expo-crypto`'s polyfill in `lib/polyfills.ts`) or include a monotonic counter. Could also be left as-is.

**‚ö† Crew sheet ‚Äî 1 P2 + 1 P3.**

### P2 ‚Äî Stale `pickedProjectId` race in template picker ‚Äî `app/(tabs)/home.tsx:569`
**Repro:** Open the project picker ‚Üí pick a project ‚Üí on the template list, swipe back to the project list (resets `pickedProjectId` to `null`) ‚Üí before the swipe completes, tap a template.
**Cause:** Template `onPress` is `() => pickedProjectId && void startInspection(pickedProjectId, t.id)`. The `pickedProjectId` check happens at press time but the closure captures whatever is in state at that moment; with React's batched state updates, the value can be stale.
**Fix:** Guard inside `startInspection` itself ‚Äî re-validate the project still exists, or pass the project id by value into a stable callback rather than closure-captured state.

## 9‚Äì12 ‚Äî Wizard Sections (Yes/No, Component Grid, Photos, Free Text)

File: [app/inspections/[id]/wizard.tsx](app/inspections/%5Bid%5D/wizard.tsx)

### P1 ‚Äî Photo picker callback bus has no handle/ID ‚Äî `lib/photoPickerBus.ts:15`
**Repro:** On a photo question, tap the photo button ‚Üí swipe back from the picker without selecting ‚Üí tap photo again on the *next* question ‚Üí select an image.
**Cause:** `setPhotoPickerCallback` overwrites a single module-level `pending` callback with no id/handle. If the first caller never resolved (because the user swiped back instead of selecting/cancelling), the second caller's setter overwrites it. The first caller's URI is fine here (it gets set to null on cancel), but the *image* selected for question 2 is delivered via the latest callback ‚Äî which is question 2's, so this case actually works. **But:** if the user on iOS receives the image-selection result while a stale `pending` from a prior cold path is still set, the resolution races. This is fragile.
**Fix:** Make `setPhotoPickerCallback` return a token; `resolvePhotoPicker` accepts the token and only fires if it matches the current pending. Also, the picker screen should call `cancelPhotoPicker()` on its unmount/back-nav to clear stale state.

### P2 ‚Äî Wizard freetext debounce flush on unmount is fire-and-forget ‚Äî `app/inspections/[id]/wizard.tsx:986-995`
**Repro:** Type into a freetext question ‚Üí swipe back to project before 500ms debounce fires ‚Üí network is offline at that moment.
**Cause:** Unmount effect calls `onCommit()` synchronously without awaiting; if the underlying `patchAnswer` rejects, the rejection is unhandled and the user's text is silently lost (it was never flushed to local storage).
**Fix:** Either await the flush before unmount completes (not always feasible in React lifecycle), or ‚Äî better ‚Äî flush to AsyncStorage synchronously before kicking off the debounce/network step, so power-off / app-kill doesn't lose the edit.

### P2 ‚Äî Step transition animations re-run on multi-step jumps ‚Äî `app/inspections/[id]/wizard.tsx:173-181`
**Repro:** Tap back arrow rapidly 5+ times.
**Cause:** Every `setStepIndex` triggers a re-animation regardless of whether `Math.abs(next - prev) > 1`. UX is sluggish, not broken.
**Fix:** If the diff is >1, jump without animation (`stepAnim.setValue(...)` instead of `Animated.timing`).

**‚ö† Wizard ‚Äî 1 P1 + 2 P2.**

> Note: Agent 2 also flagged a `harness_name` field-loss claim in the wizard's `stripServerFields` merge. **Discarded** on verification ‚Äî `stripServerFields` only strips `status`, `completed_at`, `updated_at`, `created_at`, `user_id` ([lib/offline.tsx:35-49](lib/offline.tsx)). `harness_name` is preserved correctly.

## 14 ‚Äî PDF Generation

Files: [lib/pdf.ts](lib/pdf.ts), [lib/sharePdf.ts](lib/sharePdf.ts)

### P3 ‚Äî `created_at` formatting has no fallback ‚Äî `lib/pdf.ts:57`
**Repro:** Generate PDF for a row whose `created_at` is somehow null/undefined (shouldn't happen given DB constraints ‚Äî defensive only).
**Cause:** `new Date(undefined).toLocaleDateString('ka')` returns `'Invalid Date'`. Will appear in the PDF header.
**Fix:** `const date = q.created_at ? new Date(q.created_at).toLocaleDateString('ka') : '‚Äî';`

### P3 ‚Äî `sharePdf` non-null assertion on `documentDirectory` ‚Äî `lib/sharePdf.ts:17`
**Repro:** Theoretical ‚Äî both `cacheDirectory` and `documentDirectory` undefined.
**Cause:** `(FileSystem.cacheDirectory ?? FileSystem.documentDirectory!)` ‚Äî second fallback uses `!` non-null assertion.
**Fix:** Replace with explicit guard + meaningful error: `if (!base) throw new Error('No filesystem dir available')`.

> Both are defensive-only; in practice these dirs are defined on iOS and Android.

**‚ö† PDF ‚Äî 2 P3.**

## 16 ‚Äî Settings / Profile (More tab)

File: [app/(tabs)/more.tsx](app/(tabs)/more.tsx)

### P1 ‚Äî Sign-out has no loading guard ‚Äî `app/(tabs)/more.tsx:192-193`
**Repro:** Tap "·Éí·Éê·É°·Éï·Éö·Éê" rapidly twice, or tap and immediately tap a section row.
**Cause:** `onPress={signOut}` is fire-and-forget ‚Äî no `loading`/`disabled` state. Rapid taps fire `signOut()` twice; navigation during the in-flight signOut can leave the user on a screen requiring a session that's mid-teardown.
**Fix:** Wrap with `const [signingOut, setSigningOut] = useState(false)` and disable the button while in flight. Pattern is already used elsewhere (e.g. `Button` accepts `loading`).

### P3 ‚Äî Locale code inconsistency ‚Äî `app/qualifications/index.tsx:130`
**Repro:** Cosmetic ‚Äî date formatting in qualifications uses `'ka'` while [home.tsx:679](app/(tabs)/home.tsx) uses `'ka-GE'`.
**Cause:** Inconsistent locale strings. Both work in modern Intl, but `'ka-GE'` is the canonical BCP-47 form and behaves more predictably across older devices.
**Fix:** Standardize on `'ka-GE'` everywhere. Also consider centralizing in `lib/locale.ts`.

**‚ö† More ‚Äî 1 P1 + 1 P3.**
