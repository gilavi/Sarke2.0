How to read this report

Critical breaks a core flow, loses data, or charges the wrong amount

C1

A strong password crashes the entire Registration page

Critical
Auth · Sign-up web-app/src/pages/auth/Register.tsx:11–29, 112–127 ✓ verified in source

Evidence

const getPasswordStrength = (password) => {
  let score = 0;
  if (password.length >= 8) score++;   // up to
  if (password.length >= 12) score++;  // SIX
  if (/[A-Z]/.test(password)) score++; // conditions
  if (/[a-z]/.test(password)) score++; // → score can reach 6
  if (/[0-9]/.test(password)) score++;
  if (/[^A-Za-z0-9]/.test(password)) score++;
  const levels = [ /* only 6 entries: indexes 0..5 */ ];
  return { score, ...levels[score] };   // levels[6] === undefined
};
// render:
{i < strength.score ? strength.color.replace('text-','bg-') : 'bg-neutral-200'}
//                     ^^^ strength.color is undefined → TypeError

Impact

Any password with ≥12 chars containing an uppercase, lowercase, digit and a symbol scores 6. levels only has indexes 0–5, so levels[6] is undefined, strength.color becomes undefined, and .replace() throws — React unmounts and the app falls into the (English) crash screen. There are also only 5 strength bars for 6 possible points. The users this hurts are exactly the security-conscious ones you least want to lose at sign-up.

Recommendation

Clamp the index (levels[Math.min(score, levels.length-1)]), add the 7th level (or collapse to 5 buckets), and render the correct number of bars.

C2

The app advertises ₾19/mo everywhere but the real checkout charges ₾1

Critical
Billing · Trust Subscribe.tsx:141,156 vs PaywallModal.tsx:71, landing/pricing.tsx:50, landing/home-cta.tsx:35 ✓ verified in source

Evidence

// Subscribe.tsx (the actual BOG checkout page)
<div class="...text-4xl font-bold">₾1</div> <div>თვეში</div>
<Button>გადახდა ₾1</Button>

// PaywallModal.tsx:71        → ₾19 / თვეში
// landing/pricing.tsx:50     → ₾19
// landing/home-cta.tsx:35    → ₾19 / თვეში

Impact

Every marketing surface and the in-app paywall promise ₾19/month; the page that actually creates the payment order charges ₾1. Best case it's a confusing bait-and-switch in the user's favour; worst case a user who was shown ₾19 in the paywall clicks "upgrade" and lands on a ₾1 screen and assumes the site is broken or fraudulent. (The ₾1 is a known soft-launch test rate — the bug is that it was never reconciled with the ₾19 copy, and there is no single source of truth for price.)

Recommendation

Put the price in one constant/config and render it everywhere. Ship the real price on the checkout the moment payments go live, and never let marketing copy and the charge diverge.

H3

Deletes fail silently — no error toast on Projects and all five History delete paths

High
FeedbackError handling Projects.tsx:150–153 · History.tsx:131–165 ✓ verified

H4

ProjectDetail actions give no success/error feedback — and one section's error blanks the whole page

High
FeedbackProject detail ProjectDetail/index.tsx:42–48 + all sections' onError ✓ verified (re-scoped)
const error = actionError ?? (queryError ? humanizeError(queryError) : null);
if (error) return <ErrorMessage>{error}</ErrorMessage>;   // replaces the ENTIRE page
// every section reports via onError={setActionError}

Adding/removing/uploading in any section produces no success toast. And because each section funnels failures into a single page-level actionError, a failed file-delete or signer-add replaces the whole project page with one bare error line — the user loses all context for a section-level problem. H5

The 404 page is 100% English, lives outside the app shell, and its only button points to a protected route

High
i18nNavigation pages/NotFound.tsx:68–90 · app/router.tsx:214 ✓ verified
<p>Wrong site.</p>
<p>This page is under construction - or just doesn't exist.</p>
<Link to={routes.home}>Back to Home</Link>   // /home is behind ProtectedRoute

In a Georgian-first product the catch-all page speaks only English. It renders outside AppShell, so a logged-in user who hits a bad link loses the whole sidebar/nav. A logged-out visitor's only escape — "Back to Home" — targets a protected route and just bounces them to login.

H6

The crash screen is English-only, leaks the raw error message, never logs, and never recovers

High
i18nError handling web-app/src/components/ErrorBoundary.tsx:16–36 ✓ verified
<h1>Something went wrong</h1>
<p>{this.state.error?.message || 'An unexpected error occurred'}</p>   // raw stack message to the user
<button onClick={() => window.location.reload()}>Reload Page</button>
// only getDerivedStateFromError — no componentDidCatch, no logging, no reset on navigation

At the worst possible moment the Georgian user sees English boilerplate plus a raw technical error.message. Nothing is reported anywhere (no componentDidCatch/Sentry), so you never learn it happened, and there's no navigation reset — a single crash blanks the app until a full manual reload.

Localize the copy, hide the raw message behind a "details" toggle, log to Sentry in componentDidCatch, and reset the boundary on route change.

H7

The language toggle can't persist and only ~⅓ of the UI is actually translated

High

H9

The calendar renders a full 7-column month grid on phones — ~50px cells, unreadable/untappable events

High
Responsive web-app/src/pages/Calendar.tsx:360–433 ✓ verified

The month view is a fixed 7-wide grid with no mobile breakpoint. On a phone each day is ~40–50px, event pills overflow/clip, and tapping the right day is a lottery. The calendar is essentially unusable on the device most site inspectors actually carry.

Switch to an agenda/list layout below a breakpoint, or make day cells scrollable with larger touch targets.

H10

Row actions are hover-only — the History delete button is invisible and unreachable on touch

High
ResponsiveTouch web-app/src/pages/History.tsx:208–210 ✓ verified
<div className="opacity-0 transition-opacity group-hover:opacity-100">
  <DeleteButton iconOnly .../>
</div>

The delete control only appears on :hover. Touch devices have no hover state, so on mobile the action is permanently at opacity 0 — a whole feature is invisible and undiscoverable for touch users.

H11

The order-wizard step bar overflows on mobile (6 no-wrap Georgian labels crammed into a fixed grid)

High

The fire-safety order variant has 6 steps; the labels use whitespace-nowrap inside a fixed center column, so on a phone the step bar overflows/clips and the user can't see where they are in the flow.

H12

Muted body text fails WCAG AA contrast in both themes

High
Accessibility web-app/src/index.css:138,172 + widespread text-neutral-400 ✓ verified

Six list screens flash their empty-state during background refetch (canonical loading rule violated)

Medium
Loading states Projects.tsx:178 · History.tsx:57,174 · Calendar.tsx:235,351 · Certificates · Qualifications · Home · AsyncBoundary.tsx:95–105 ✓ verified
{isLoading && <SkeletonList/>}
